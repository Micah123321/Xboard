## 1. 需求

### 背景
订单管理页已有单条“确认佣金”能力，但管理员处理多笔待确认邀请佣金时需要逐条打开操作菜单确认，效率低且容易遗漏。仪表盘和订单页已支持“真实待确认佣金订单”筛选，因此批量确认应建立在当前待确认记录规则上，而不是放宽佣金状态流转。

### 目标
新增管理端“批量同意邀请佣金”能力：管理员勾选订单后，可一次性把符合条件的待确认佣金订单从 `commission_status=0` 更新为 `commission_status=1`，不符合条件的记录自动跳过，并在前端提示已确认与跳过数量。

### 约束条件
```yaml
时间约束: 本轮完成后端接口、前端接入与本地验证
性能约束: 批量接口一次只处理当前选中的订单 ID，不扫描全表
兼容性约束: 保留现有单条 /order/update 行为和订单列表筛选行为
业务约束: 仅处理已完成订单、真实佣金金额大于 0、存在邀请人、佣金状态为待确认的记录
```

### 验收标准
- [ ] 后端提供管理员批量确认接口，返回 `confirmed`、`skipped`、`total` 与 `skipped_ids`。
- [ ] 后端只把符合条件的待确认佣金订单更新为发放中，不改变已发放、无效、无佣金或未完成订单。
- [ ] 管理端订单列表支持多选订单，并提供“批量确认佣金”按钮。
- [ ] 前端批量确认完成后刷新列表，清空本页选择，并以 toast 展示确认与跳过数量。
- [ ] 现有单条确认佣金、佣金筛选、订单详情佣金状态维护不回归。

## 2. 方案

### 技术方案
在后端 `OrderController` 新增 `batchConfirmCommission()`，接收订单 `ids`，先限定 ID 集合，再使用与前端 `canQuickConfirmCommission()` 一致的业务条件筛选可确认订单：`status=3`、`commission_status=0`、`invite_user_id` 非空、`commission_balance>0`。接口只更新命中记录为 `commission_status=1`，返回统计结果和跳过 ID。

在 `AdminRoute` 注册 `POST /order/batchConfirmCommission`。前端 `admin.ts` 新增 API 封装和响应类型，`OrdersView.vue` 启用 Element Plus 表格多选，维护 `selectedOrders` 与可确认数量，工具条新增“批量确认佣金”按钮；提交前弹窗确认，提交后根据响应展示“已确认 N 笔，跳过 M 笔”，刷新列表并重置选择。

### 影响范围
```yaml
涉及模块:
  - 后端订单管理: 新增批量确认佣金接口与路由
  - 管理端 API: 新增批量确认佣金请求封装与类型
  - 管理端订单页: 新增表格多选、批量按钮、批量提交状态和反馈
  - 知识库: 更新订单管理上下文、CHANGELOG 和方案包归档记录
预计变更文件: 7-9
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 佣金状态属于资金相关流程，误改非待确认记录会影响发放链路 | 中 | 后端强制条件筛选，只允许 `commission_status=0` 到 `1`，其他记录跳过 |
| 前端选择跨分页记录时状态可能已变化 | 低 | 后端以当前数据库状态为准，返回跳过数量，前端不假设全成功 |
| 表格多选影响现有布局宽度 | 低 | 使用 Element Plus 原生 selection 列，保持现有 Apple 化表格样式 |

### 方案取舍
```yaml
唯一方案理由: 新增后端批量接口能保证状态流转由服务端统一校验，避免前端逐条调用造成部分失败难以汇总、并发请求过多和规则分散。
放弃的替代路径:
  - 前端逐条调用现有 /order/update: 规则仍在前端筛选，无法返回准确跳过统计，且批量操作时请求数随订单数线性增加。
  - 允许强制确认所有勾选记录: 会突破用户已确认的范围边界，存在误改已发放/无效/无佣金订单风险。
回滚边界: 可独立撤销新路由、新控制器方法、新 API 封装和订单页多选批量按钮；不涉及数据库结构变更。
```

## 3. 技术设计

### API设计
#### POST `/api/v2/{secure_path}/order/batchConfirmCommission`
- **请求**:
```json
{
  "ids": [1, 2, 3]
}
```
- **响应**:
```json
{
  "confirmed": 2,
  "skipped": 1,
  "total": 3,
  "skipped_ids": [3]
}
```

### 数据模型
不新增表和字段。仅更新 `v2_order.commission_status`，状态含义沿用现有模型：`0=待确认`、`1=发放中`、`2=已发放`、`3=无效`。

## 4. 核心场景

### 场景: 批量确认待确认佣金
**模块**: order-payment
**条件**: 管理员在订单管理页勾选多笔订单，其中部分订单满足待确认佣金条件。
**行为**: 点击“批量确认佣金”并确认操作。
**结果**: 满足条件的订单佣金状态更新为发放中，不满足条件的订单跳过；页面刷新并提示确认/跳过数量。

---

## 5. 技术决策

### batch-approve-invite-commission#D001: 批量确认必须由后端统一筛选
**日期**: 2026-05-08
**状态**: ✅采纳
**背景**: 批量同意邀请佣金会改变资金发放流程状态，不能只依赖前端按钮可见性判断。
**选项分析**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 新增后端批量确认接口 | 规则集中、统计准确、请求少、可审计 | 需要新增接口和类型 |
| B: 前端循环调用现有单条接口 | 改动少 | 规则分散、请求多、跳过统计不准确 |
**决策**: 选择方案 A
**理由**: 佣金状态变更应由后端作为最终守门点，前端仅负责选择和反馈。
**影响**: 后端订单接口新增批量端点，管理端订单页改为支持选中记录批量确认。

## 7. 成果设计

### 设计方向
- **美学基调**: 延续 Xboard 管理端 Apple 化运营后台，低噪音白色工作台、蓝色主操作、表格密度稳定。
- **记忆点**: 表格底部和工具条都能明确看到当前选择数量，批量按钮只在可执行时突出。
- **参考**: `.helloagents/DESIGN.md` 现有订单管理页约束。

### 视觉要素
- **配色**: 继续使用 `#0071e3` 作为主操作色，跳过/警告信息使用现有 warning 语义色。
- **字体**: 复用项目既有后台字体策略，不引入新字体。
- **布局**: 工具条右侧加入批量按钮，表格左侧加入原生 selection 列，底部选择数量从静态文案改为真实数量。
- **动效**: 复用 Element Plus loading、confirm 和 toast 反馈。
- **氛围**: 不新增装饰，保持订单工作台的运营效率。

### 技术约束
- **可访问性**: 批量按钮提供禁用态，确认弹窗明确本次将处理的数量。
- **响应式**: 工具条沿用现有换行策略，窄屏下批量按钮进入右侧按钮组换行显示。
